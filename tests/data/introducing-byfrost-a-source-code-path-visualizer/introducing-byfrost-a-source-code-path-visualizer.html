Introducing Byfrost-A Source Code Path Visualizer

Okay so the name might change as time goes on but hear me out!

It’s been a long time coming reader. The elusive indexing tool I’ve been attemtping to build for so long is finally upon us. What I hesitate to call It’s a cute little idea I’ve been mulling over for a while and figured it’s high time I finally took a stab at pulling it off. My power level’s risen significantly over the past few months too; my language parsing and compiler skills have improved (probably). How difficult could this really actually be, right?

So without further ado, I introduce for to you for your appraisal; the Byfrost Indexer!

Code Indexing — What is this exactly?

I loosely call it this because I can’t think of anything more descriptive. The core project revolves around indexing a codebase; grabbing all its functions, variables and everything that makes up the codebase and organizing it in a conrollable structure of somekind.

Every IDE does this pretty well — the indexing part I mean; GoLand, VSCode, IntelliJ, Visual Studio among others. Some command line utilities do this incredibly well too, OSX’s ag utility comes to mind. And for most use cases for the everyday developer makin’ their way down town, these tools suffice pretty damn well. I myself perfer to use ag for all codebase searches because I too like to live dangerously, you see.

Which segues nicely into the crux of the matter and why I’ve been dreaming up the ultimate codebase indexing tool for the most epic developer experience for so very long.

Code Indexing — What is yours exactly?

The problem goes like this; learning a new codebase and finding where things are is a bit of an artform. From hacking together a bunch of indexing techniques to committing a control flow you’re going down to memory, to hopping between various files in your code editor to get a sense of what a program is doing, things can get pretty out of hand, especially if the code you’re reading is large, undocumented and a bit all over the place.

So then I had an idea…

I’m reminded of the pretty great reverse engineering tools and how they go about visualizing the branching flows of binary files with some runtime data on variable information thrown in for great value for money 🌠. This made me wonder; why wasn’t this done for source code files. 🤔? As engineers, we spend a lot more time reading code than writing it and my hacky approach of ag-ing it worked pretty well for me but it got to me quite a bit when I would start out trying to contribute to opensource or when starting out at a new job and understanding the control flow of everything felt like pulling teeth.

I found a few good tools for this; Sourcetrail being one of them but I wanted something even easier to work with. Something that would make reading code as intuitive and as simple as reading… well… reading a book really. The ultimate reading experience!

I couldn’t think of why this was a terrible idea so I’ve decided to build my own!

Introducing the Byfrost Indexer!!

Behold!! The indexer at its finest!

So what is Byfrost? My unique spin on the source code indexing experience. I wanted something that both felt cool to use and relatively straightforward to intutiviely understand. I also wanted something I could use with my IDE (I don’t know how this is gonna work yet but as a plugin, this would be pretty sick as an assist).

So a few core ideas I settled on for this design:

  • No files — The idea is to solely focus on reading the path various functions and data structures as they are used, abstracting everything else in the those files away. But if I want to go to that specific function, there’s a file path and name just above the block of code that will open specific location in your editor when clicked.
  • As little extras as possible — I feel when exploring a codebase, I’m often exploring for a very specific reasons. To make this as intuitive as possible, I decided to keep it to just exploring a code path statically as live tracing for a first version is overkill and kinda sorta really hard. So all that works is being able to read a block of code and clicking on the symbol to expand (like turning the pages of an epic, albeit poorly written novel! 😃)
  • Collaboration — The last detail is keeping to the idea of collaborating (specifically communicating) across multiple engineers. It’s easier to present and talk about a large codebase in collaborative settings like on Google Meets when you’re able to show the flow of a program (and your thought processes) to other engineers.

The three core ideas above drive my thought process for the first version of my super awesome indexing tool. But also, I wanted to make something that looks and feels super cool to use too. My frontend skills aren’t up to scratch yet but they’re getting there.

Okay But Why Byfrost?

It’s a Psycho-Pass reference. Trust me, the name will make sense when you see the reference, I promise! It’s a cool reference!

And that’s it. This is my new new project I’ll be working on for a long while. It’s nowhere near ready for primetime but I wanted to get first impressions on this concept from other developers and get a sense of what y’all think of this kind of codebase reading and indexing approach.

Yup, see you soon!
